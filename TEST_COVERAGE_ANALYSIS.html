Recommended Testing Strategy

Extract pure business logic functions into separate .js modules and test with a lightweight runner (Vitest or Jest). UI flows can later be covered with Playwright E2E tests.

Priority Areas for Test Coverage

P1 Critical Business Logic — High Impact, Easy to Extract

1.1 Date & Delivery Validation

Files: 에이엠티.html, 조혼가구.html

Functions to test:
  • isWeekendOrHoliday(date) — Checks weekends + Korean holiday list
  • getInitialDeliveryDate() — Default delivery date (today + 4 business days)
  • validateDeliveryDate(el) — Adjusts user-selected dates to valid business days
  • _workDayOffset(dateStr, n) — Offsets a date by N work days
  • getKSTDateStr(offsetDays) — KST-aware date string generation
Why: Date logic bugs silently cause incorrect delivery promises.
Suggested tests (~15-20 cases):
  • Weekend dates get pushed to Monday
  • Korean holidays are skipped correctly
  • Year-end boundary cases (Dec 31 → Jan holidays)
  • Leap year handling
  • Timezone edge cases (KST offset)

1.2 Data Normalization & Product Identity

Files: 에이엠티.html, 조혼가구.html

Functions to test:
  • normalizeValue(val) — Normalizes null/undefined/empty/"선택없음" values
  • buildDisplayName(v1, v2, v3, v4) — Constructs product display names from 4 variant fields
  • buildProductIdentities() — Builds unique product identity lookup structures
Why: Incorrect normalization leads to duplicate products, wrong orders, or display errors.
Suggested tests (~10-15 cases):
  • null, undefined, "", " ", "선택없음" all normalize to "선택없음"
  • Display names concatenate correctly with missing variants
  • Product identity uniqueness is maintained

1.3 Process Time Calculation

File: 조혼가구.html

Function: calcProcessTime(row, qty, difficulties, h) — Calculates manufacturing process time based on product attributes, quantity, difficulty factors, and hours.
Why: Incorrect process time directly impacts production scheduling and delivery promises.
Suggested tests (~5-10 cases):
  • Base case calculations with known inputs
  • Edge cases: quantity = 0, missing difficulty values
  • Rounding behavior

P2 Data Processing — Medium Impact, Moderate Effort

2.1 Excel File Parsing

Files: 에이엠티.html, 조혼가구.html

Function: handleFileUpload(input) — Parses uploaded Excel files via SheetJS and builds internal data structures.
Why: Primary data ingestion path. Malformed Excel files could crash the app or corrupt data silently.
Suggested tests (~10-15 cases):
  • Valid Excel file parses correctly
  • Missing columns handled gracefully
  • Empty sheets don't crash
  • Large files (1000+ rows) process correctly

2.2 Order Management

File: 에이엠티.html

Functions to test:
  • addOrderedProduct() — Adds products with validation
  • confirmSelectedToTimeline() — Moves checked orders to confirmed timeline
  • moveToRegistered(docId) — Transitions order status in Firestore
  • _buildPriorityMap(items) — Builds priority ordering for cutting/production
Why: Order state transitions are the core business workflow. Wrong transitions can cause lost or duplicate orders.
Suggested tests (~15-20 cases):
  • Cannot add order with missing required fields
  • Quantity validation (positive integers only)
  • State transitions follow valid paths
  • Priority map ordering is deterministic

2.3 Search & Filtering

Files: 에이엠티.html, 조혼가구.html

Functions to test:
  • handleSearch(query) — Fuzzy product search across multiple fields
  • Category cascade: onSupplierChange() → onCodeChange() → onL1Change() → ... → onL4Change()
Why: Search bugs prevent users from finding products, directly impacting order volume.
Suggested tests (~10-15 cases):
  • Search matches partial strings
  • Empty search returns all results
  • Category filters cascade correctly (selecting L1 resets L2-L4)

P3 Firebase Integration — High Impact, Higher Effort

3.1 Real-time Sync (all apps)

Functions:
  • initCloudDataSync() — Sets up Firestore real-time listeners
  • initConfirmedOrdersSync() — Syncs confirmed orders across devices
  • loadCloudExcelData(url, fileName) — Downloads and parses Excel from Firebase Storage
Why: Sync failures can cause data loss or stale data without user awareness.
Approach: Use Firebase Local Emulator Suite for integration tests.

3.2 Notification System (에이엠티)

Functions:
  • addNotification(type, item) — Creates in-app notifications
  • handleNotificationAction(docId, type, notificationId) — Processes notification actions
  • clearAllNotifications()
Why: Missed notifications mean missed order changes or approvals.

P4 UI Component Behavior — E2E Tests

  • Cutting Board: renderID0508(), setCuttingSort(type)
  • Timeline: renderTimelineConfirmedList(), toggleTlConfDetails(idNum)
  • QR Scanning: Camera initialization, barcode parsing (반품스캔)
Approach: Better suited for Playwright/Cypress E2E tests rather than unit tests, since they depend heavily on DOM state.

Recommended Implementation Plan

  1. Set up test infrastructure: npm init -y, install Vitest + jsdom
  2. Extract pure functions: Create src/utils/dates.js, products.js, process.js, search.js
  3. Write P1 unit tests (~30-45 cases): Date validation, normalization, process time
  4. Add Firebase Emulator tests (P3): Firestore sync, auth flows, storage
  5. Add Playwright E2E tests (P4): Critical user flows end-to-end
